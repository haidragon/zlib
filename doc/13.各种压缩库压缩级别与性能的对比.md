# 12. 各种压缩库压缩级别与性能的对比

## 一、概述

zlib/libz 提供了 10 个压缩级别 (0-9)，每个级别在压缩速度、压缩率和内存占用之间做出不同的权衡。本文档深入分析这些级别的源码实现，并与其他主流压缩库进行性能对比。

## 二、zlib 压缩级别详解

### 2.1 压缩级别配置表

```c
// deflate.c:117-130
typedef struct config_s {
   unsigned short good_length; /* reduce lazy search above this match length */
   unsigned short max_lazy;    /* do not perform lazy search above this match length */
   unsigned short nice_length; /* quit search above this match length */
   unsigned short max_chain;   /* 最大哈希链长度 */
   compress_func func;         /* 压缩函数指针 */
} config;

static const config configuration_table[10] = {
/*      good lazy nice chain */
/* 0 */ {0,    0,  0,    0, deflate_stored},  /* store only - 仅存储不压缩 */
/* 1 */ {4,    4,  8,    4, deflate_fast},   /* max speed - 最快速度 */
/* 2 */ {4,    5, 16,    8, deflate_fast},   
/* 3 */ {4,    6, 32,   32, deflate_fast},   

/* 4 */ {4,    4, 16,   16, deflate_slow},   /* lazy matches - 延迟匹配 */
/* 5 */ {8,   16, 32,   32, deflate_slow},   
/* 6 */ {8,   16, 128, 128, deflate_slow},   /* default - 默认级别 */
/* 7 */ {8,   32, 128, 256, deflate_slow},   
/* 8 */ {32, 128, 258, 1024, deflate_slow},  
/* 9 */ {32, 258, 258, 4096, deflate_slow}}; /* max compression - 最大压缩 */
```

### 2.2 参数含义解析

#### good_length (最佳匹配长度)
- **作用**: 当找到超过此长度的匹配时，减少后续搜索
- **影响**: 较大的值会增加压缩时间但提高压缩率
- **范围**: 0-32

#### max_lazy (最大延迟搜索长度)
- **作用**: 延迟匹配的最大尝试长度
- **原理**: 如果当前匹配长度为 L，继续搜索是否有更长的匹配
- **影响**: 
  - 小值 → 快速但压缩率低
  - 大值 → 慢速但压缩率高
- **范围**: 0-258 (MAX_MATCH)

#### nice_length (理想长度)
- **作用**: 找到超过此长度的匹配后立即停止搜索
- **优化**: 避免无谓的搜索，平衡速度与压缩率
- **范围**: 8-258

#### max_chain (最大哈希链长度)
- **作用**: 哈希冲突链的最大遍历长度
- **影响**: 
  - 小值 → 快速找到第一个匹配，可能错过更好匹配
  - 大值 → 搜索更多候选，找到更长匹配
- **范围**: 4-4096

### 2.3 压缩算法选择

#### deflate_fast (级别 1-3)

```c
// deflate.c:700-850
local block_state deflate_fast(deflate_state *s, int flush)
{
    IPos hash_head;       /* head of hash chain */
    int bflush;           /* set if current block must be flushed */
    
    for (;;) {
        // 检查输入是否用完
        if (s->strstart >= s->lookahead) {
            fill_window(s);
            if (s->strstart >= s->lookahead)
                break;
        }
        
        // 插入字符串到哈希表
        INSERT_STRING(s, s->strstart, hash_head);
        
        // 查找最长匹配
        s->match_length = MIN_MATCH - 1;  // 初始化为最小匹配 -1
        
        if (hash_head != NIL && 
            s->strstart - hash_head <= MAX_DIST(s)) {
            // 在限制距离内，查找匹配
            s->match_length = longest_match(s, hash_head);
        }
        
        // deflate_fast 策略：不进行延迟匹配
        // 直接输出找到的匹配
        if (s->match_length >= MIN_MATCH) {
            _tr_tally_dist(s, s->strstart - s->match_start,
                          s->match_length - MIN_MATCH, bflush);
            s->lookahead -= s->match_length;
            s->strstart += s->match_length;
            s->match_length = 0;
        } else {
            // 无匹配，输出字面量
            _tr_tally_lit(s, s->window[s->strstart], bflush);
            s->lookahead--;
            s->strstart++;
        }
        
        if (bflush) FLUSH_BLOCK(s, 0);
    }
    
    FLUSH_BLOCK(s, flush == Z_FINISH);
    return block_done;
}
```

**特点**:
- 找到第一个可用匹配就使用
- 不尝试寻找更优匹配
- 适合速度敏感场景

#### deflate_slow (级别 4-9)

```c
// deflate.c:852-1050
local block_state deflate_slow(deflate_state *s, int flush)
{
    IPos hash_head;       /* head of hash chain */
    int bflush;           /* set if current block must be flushed */
    
    for (;;) {
        // 检查输入
        if (s->strstart >= s->lookahead) {
            fill_window(s);
            if (s->strstart >= s->lookahead)
                break;
        }
        
        // 插入哈希
        INSERT_STRING(s, s->strstart, hash_head);
        
        // 查找匹配
        s->match_length = MIN_MATCH - 1;
        
        if (hash_head != NIL && 
            s->prev_length < s->max_lazy &&
            s->strstart - hash_head <= MAX_DIST(s)) {
            
            if (longest_match(s, hash_head)) {
                // 找到匹配
            }
        }
        
        // === 延迟匹配策略 ===
        if (s->prev_length >= MIN_MATCH && 
            s->match_length <= s->prev_length) {
            
            // 上一位置的匹配更好，输出上一匹配
            _tr_tally_dist(s, s->strstart - 1 - s->prev_match,
                          s->prev_length - MIN_MATCH, bflush);
            
            // 跳过已匹配的字符
            s->lookahead -= s->prev_length - 1;
            s->prev_length = 0;
            s->match_available = 0;
            s->strstart += s->prev_length - 1;
            
        } else {
            // 当前匹配更好，或无匹配
            if (s->prev_length >= MIN_MATCH) {
                // 输出一系列字面量代替上一匹配
                _tr_tally_lit(s, s->window[s->strstart - 1], bflush);
            }
            
            s->prev_length = s->match_length;
            s->match_available = 1;
            s->strstart++;
        }
        
        if (bflush) FLUSH_BLOCK(s, 0);
    }
    
    // 处理最后的延迟匹配
    if (s->match_available) {
        _tr_tally_lit(s, s->window[s->strstart - 1], bflush);
        s->match_available = 0;
    }
    
    FLUSH_BLOCK(s, flush == Z_FINISH);
    return block_done;
}
```

**延迟匹配原理**:
```
输入数据：a b c d e f g h i j ...
位置 N:   找到匹配 "abcd" (长度 4)
位置 N+1: 找到匹配 "bcdefg" (长度 6)

延迟匹配决策:
- 在位置 N 时，先不输出
- 移动到 N+1，发现更好的匹配
- 回溯输出 N 的字面量 'a'
- 输出 N+1 的长匹配 "bcdefg"

优势：用单个字符代价换取更长匹配
```

### 2.4 deflate_stored (级别 0)

```c
// deflate.c:570-650
local block_state deflate_stored(deflate_state *s, int flush)
{
    // 不压缩，直接存储原始数据
    
    while (s->lookahead > 0) {
        // 计算块大小 (最大 64KB - 5)
        unsigned int len = s->lookahead;
        if (len > s->max_block_size)
            len = s->max_block_size;
        
        // 复制原始数据
        memcpy(s->pending_buf + s->pending, 
               s->window + s->strstart, 
               len);
        
        // 发送存储块头
        _tr_stored_block(s, (char*)s->window + s->strstart, 
                        len, 0);
        
        s->strstart += len;
        s->lookahead -= len;
    }
    
    FLUSH_BLOCK(s, flush == Z_FINISH);
    return finish_done;
}
```

**应用场景**:
- 数据已经高度压缩
- 需要最快处理速度
- 网络传输中动态切换压缩

## 六、实际应用场景推荐

### 6.1 网络传输 (HTTP 压缩)

**推荐**: zlib 级别 1-3 或 zstd

```c
// Nginx gzip 配置示例
gzip_comp_level 2;  # 级别 2，平衡 CPU 和带宽

// 或使用 zlib-stream
deflateInit(&strm, 1);  // 最快压缩，减少延迟
```

**理由**:
- 低延迟要求
- 解压速度快
- 节省带宽明显

### 6.2 日志文件归档

**推荐**: zlib 级别 6 或 zstd 级别 3

```bash
# gzip 默认级别 6
gzip -6 logfile.txt

# 或使用 zstd 获得更好性能
zstd -3 logfile.txt
```

**理由**:
- 批量处理，时间不敏感
- 需要较好的压缩率
- 兼容性要求高

### 6.3 嵌入式系统固件

**推荐**: zlib 级别 9

```c
// 一次性压缩，多次解压
deflateInit(&strm, 9);  // 追求最小体积
```

**理由**:
- ROM 空间宝贵
- 压缩一次，解压多次
- 对压缩时间不敏感

### 6.4 实时通信 (VoIP、游戏)

**推荐**: lz4 或 zlib 级别 1

```c
// 最低延迟
deflateInit(&strm, 1);
// 或使用 lz4
```

**理由**:
- 微秒级延迟要求
- 宁可牺牲压缩率
- 高速吞吐

### 6.5 数据库备份

**推荐**: zlib 级别 6-8 或 zstd 级别 5-7

```bash
# mysqldump 管道压缩
mysqldump | gzip -7 > backup.sql.gz

# 或使用 pigz (并行 gzip)
mysqldump | pigz -6 > backup.sql.gz
```

**理由**:
- 批量处理
- 存储空间考虑
- 可接受较长压缩时间

### 6.6 移动应用资源

**推荐**: zlib 级别 9 或 Brotli

```bash
# Android APK 资源压缩
zip -9 -r app.apk assets/

# Web 资源使用 Brotli
broccoli -q 11 style.css
```

**理由**:
- 减少下载流量
- 安装后解压
- 用户体验优先

## 七、性能优化技巧

### 7.1 选择合适的窗口大小

```c
// 小数据块：减小窗口
deflateInit2(&strm, 6, Z_DEFLATED, 
             12,  // 4KB 窗口 (而非 32KB)
             8, Z_DEFAULT_STRATEGY);

// 大数据流：保持 32KB 窗口
deflateInit(&strm, 6);  // 默认 15 bits = 32KB
```

**效果**:
- 小窗口：节省内存 50KB，压缩率降低 5-10%
- 大窗口：压缩率提高，尤其是大文件

### 7.2 使用策略标志

```c
// 文本数据
deflateInit2(&strm, 6, Z_DEFLATED, 15, 8, 
             Z_DEFAULT_STRATEGY);

// 已压缩数据 (如 JPEG inside ZIP)
deflateInit2(&strm, 0, Z_DEFLATED, 15, 8, 
             Z_STORED);  // 直接存储

// 过滤数据 (PNG 图像)
deflateInit2(&strm, 6, Z_DEFLATED, 15, 8, 
             Z_FILTERED);  // 优化小匹配

// Huffman 专用
deflateInit2(&strm, 6, Z_DEFLATED, 15, 8, 
             Z_HUFFMAN_ONLY);  // 仅 Huffman，无 LZ77

// RLE 模式 (简单重复)
deflateInit2(&strm, 6, Z_DEFLATED, 15, 8, 
             Z_RLE);  // 游程编码
```

### 7.3 分块压缩策略

```c
// 错误做法：一次性压缩大文件
compress(large_file);  // 内存占用高，无法流式处理

// 正确做法：分块处理
#define CHUNK_SIZE 16384

while (has_input()) {
    read_chunk(input, CHUNK_SIZE);
    deflate(&strm, Z_NO_FLUSH);  // 持续压缩
    
    if (output_buffer_full()) {
        write_output();
        reset_output_buffer();
    }
}

deflate(&strm, Z_FINISH);  // 结束压缩
```

**优势**:
- 内存占用恒定
- 支持流式处理
- 可中断恢复

### 7.4 并行压缩

```bash
# 使用 pigz (并行 gzip)
pigz -6 -p 4 largefile.txt  # 4 线程

# 内部自动检测 CPU 核心
pigz -6 largefile.txt
```

**性能提升**:
- 4 核 CPU: 3.5x 加速
- 8 核 CPU: 6.8x 加速

## 八、压缩级别选择决策树

```
开始
│
├─ 是否需要最快速度？
│  ├─ 是 → 级别 0 (存储) 或 级别 1 (最快)
│  └─ 否 ↓
│
├─ 是否是实时应用？(网络、通话)
│  ├─ 是 → 级别 1-2
│  └─ 否 ↓
│
├─ 是否每日备份/归档？
│  ├─ 是 → 级别 6 (默认)
│  └─ 否 ↓
│
├─ 是否长期保存？(>1 年)
│  ├─ 是 → 级别 8-9 或 xz/bzip2
│  └─ 否 ↓
│
├─ 是否嵌入式设备？
│  ├─ 是 → 级别 9 (最小体积)
│  └─ 否 ↓
│
└─ 默认推荐 → 级别 4-6 (平衡)
```

### 9.2 边际效益分析

**压缩级别 vs 收益**:

```

    │                  ╱

    │      ╱
```

- 级别 1-3: 性价比最高区间
- 级别 4-6: 平衡点
- 级别 7-9: 边际效益递减

## 十、总结与建议

### 10.1 快速参考表

| 需求 | 推荐级别 | 替代方案 |
|------|----------|----------|
| 最快速度 | 0 或 1 | lz4 |
| 低延迟网络 | 1-2 | zstd -1 |
| 日常压缩 | 4-6 | zstd -3 |
| 高压缩率 | 8-9 | xz -9 |
| 最小体积 | 9 | bzip2 -9 |
| 兼容性好 | 6 | gzip -6 |
| 嵌入式 | 6-9 | - |
| 多媒体 | 0 (不压缩) | - |

### 10.2 黄金法则

1. **不要盲目使用级别 9**: 除非确实需要那 2% 的提升
2. **级别 6 是最佳起点**: 适用于 80% 场景
3. **已压缩文件不要再压缩**: 浪费 CPU 周期
4. **考虑解压端**: 压缩一次，解压多次的场景值得高级别
5. **测试你的数据**: 不同类型数据表现差异巨大

### 10.3 未来趋势

- **zstd 正在取代 zlib**: 更好的压缩率+速度组合
- **硬件加速**: Intel QAT 等专用压缩芯片
- **自适应压缩**: 根据内容动态调整策略
- **并行化**: 多核 CPU 利用率持续提升

## 附录 A: 测试脚本

```bash
#!/bin/bash
# benchmark.sh - 压缩性能测试脚本

FILE="testfile.dat"
LEVELS=(0 1 3 6 9)

echo "File: $FILE ($(stat -f%z "$FILE" 2>/dev/null || stat -c%s "$FILE") bytes)"
echo ""

for level in "${LEVELS[@]}"; do
    echo "=== Level $level ==="
    
    # 压缩
    START=$(date +%s.%N)
    gzip -$level -c "$FILE" > "test.gz"
    END=$(date +%s.%N)
    
    COMPRESSED=$(stat -f%z "test.gz" 2>/dev/null || stat -c%s "test.gz")
    TIME=$(echo "$END - $START" | bc)
    
    echo "Compressed: $COMPRESSED bytes"
    echo "Time: ${TIME}s"
    
    # 解压
    START=$(date +%s.%N)
    gunzip -c "test.gz" > /dev/null
    END=$(date +%s.%N)
    
    DECOMP_TIME=$(echo "$END - $START" | bc)
    echo "Decompress: ${DECOMP_TIME}s"
    echo ""
done
```
