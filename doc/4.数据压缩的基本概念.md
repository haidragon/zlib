# 数据压缩的基本概念

## 一、压缩概述

### 1.1 什么是数据压缩

数据压缩是一种通过减少数据中的冗余信息来减小数据大小的技术，其目的是：

- **节省存储空间**: 减少数据占用的磁盘或内存空间
- **减少传输时间**: 在网络传输中减少带宽消耗和传输时间
- **提高效率**: 加快数据处理和访问速度

### 1.2 压缩分类

#### 按信息损失分类

```
┌─────────────────────────────────────────┐
│           数据压缩                        │
├──────────────┬──────────────────────────┤
│   无损压缩    │        有损压缩           │
├──────────────┼──────────────────────────┤
│ • 数据可100%  │ • 牺牲部分数据精度        │
│   还原        │ • 压缩率更高              │
│ • 文本、程序   │ • 图像、音频、视频        │
│ • zlib/zip   │ • JPEG, MP3, H.264      │
└──────────────┴──────────────────────────┘
```

#### 按压缩算法分类

1. **字典编码**: LZ77, LZ78, LZW
2. **统计编码**: 霍夫曼编码, 算术编码
3. **变换编码**: 离散余弦变换(DCT), 小波变换
4. **预测编码**: DPCM, ADPCM

### 1.3 压缩性能指标

#### 压缩率 (Compression Ratio)
```
压缩率 = 原始数据大小 / 压缩后数据大小
压缩比 = (1 - 压缩后大小 / 原始大小) × 100%
```

例如：
- 原始数据: 1000字节
- 压缩后: 250字节
- 压缩率: 4:1
- 压缩比: 75%

#### 其他指标
- **速度**: 压缩/解压时间
- **内存占用**: 压缩过程所需的内存
- **复杂度**: 算法实现和计算复杂度
- **延迟**: 流式处理的延迟

## 二、熵与信息论基础

### 2.1 熵 (Entropy)

熵是信息论中衡量信息量的基本概念，由克劳德·香农提出。

#### 定义
对于一个离散随机变量X，其熵H(X)定义为：
```
H(X) = -Σ p(x) × log₂p(x)
```
其中：
- p(x) 是符号x出现的概率
- 单位是比特(bit)

#### 示例计算
假设一个字符串由4个字符组成：A, B, C, D

**情况1：均匀分布**
```
P(A) = P(B) = P(C) = P(D) = 0.25
H = -4 × (0.25 × log₂0.25) = -4 × (0.25 × -2) = 2 bits/符号
```

**情况2：非均匀分布**
```
P(A) = 0.5, P(B) = 0.25, P(C) = 0.125, P(D) = 0.125
H = -0.5×log₂0.5 - 0.25×log₂0.25 - 0.125×log₂0.125 - 0.125×log₂0.125
  = -0.5×(-1) - 0.25×(-2) - 0.125×(-3) - 0.125×(-3)
  = 0.5 + 0.5 + 0.375 + 0.375
  = 1.75 bits/符号
```

**结论**: 分布越不均匀，熵越小，压缩潜力越大。

### 2.2 信息量

单个事件的信息量：
```
I(x) = -log₂p(x)
```

含义：
- 概率越小的事件，信息量越大
- 概率为1的事件（必然事件），信息量为0

### 2.3 信源编码定理

香农第一定理（无失真信源编码定理）：
```
对于熵为H(X)的信源，无法用少于H(X) bits/符号的平均码长进行编码
```

实际编码的平均码长L满足：
```
H(X) ≤ L < H(X) + 1
```

### 2.4 条件熵与联合熵

#### 条件熵
```
H(Y|X) = -Σ p(x,y) × log₂p(y|x)
```
表示在已知X的情况下，Y的不确定性。

#### 联合熵
```
H(X,Y) = H(X) + H(Y|X) = H(Y) + H(X|Y)
```
表示两个变量的总不确定性。

#### 互信息
```
I(X;Y) = H(X) + H(Y) - H(X,Y)
```
表示两个变量之间的共享信息量。

## 三、数据中的冗余类型

### 3.1 统计冗余

符号出现的概率不均匀。例如英文文本中：
- 字母'e'出现频率约12.7%
- 字母'z'出现频率约0.07%

**压缩方法**: 霍夫曼编码、算术编码

### 3.2 上下文冗余

符号之间存在相关性。例如：
- "the"后面常跟空格或元音字母
- 'q'后面几乎总是'u'

**压缩方法**: 上下文建模、预测编码

### 3.3 词典冗余（结构冗余）

重复出现的字符串或模式。例如：
- 源代码中的变量名
- 文档中的词汇
- 重复的短语

**压缩方法**: LZ77, LZ78, LZW字典编码

### 3.4 编码冗余

符号的编码长度不是最优的。例如：
- ASCII编码：每个字符固定8位
- 即使'a'出现1000次，'z'只出现1次，都用8位编码

**压缩方法**: 变长编码（霍夫曼编码）

### 3.5 空间冗余

相邻元素之间高度相关。例如：
- 图像中相邻像素颜色相似
- 音频中相邻采样值变化不大

**压缩方法**: 变换编码、预测编码

## 四、无损压缩算法原理

### 4.1 霍夫曼编码 (Huffman Coding)

#### 基本原理
- 统计符号出现频率
- 为频繁出现的符号分配短编码
- 为不频繁出现的符号分配长编码
- 编码是前缀码，无歧义

#### 构建步骤

1. **统计频率**: 统计每个符号的出现次数
2. **构建队列**: 将每个符号作为独立的树节点，按频率排序
3. **构建霍夫曼树**:
   - 取出频率最小的两个节点
   - 创建父节点，频率为两者之和
   - 将父节点放回队列
   - 重复直到只剩一个节点
4. **生成编码**: 左子节点编码加'0'，右子节点编码加'1'

#### 示例

字符串: "ABRACADABRA"

```
步骤1: 统计频率
A: 5, B: 2, R: 2, C: 1, D: 1

步骤2-3: 构建霍夫曼树
      (11)
     /    \
   A(5)   (6)
        /    \
      R(2)   (4)
            /    \
          B(2)   (2)
               /    \
             C(1)   D(1)

步骤4: 生成编码
A: 0
R: 10
B: 110
C: 1110
D: 1111

编码结果:
ABRACADABRA = 0 110 10 0 1110 0 1111 0 110 10 0
            = 01101001110011110110100
            = 23 bits (原始: 11 × 8 = 88 bits)
            压缩率: 88/23 ≈ 3.8:1
```

#### 优缺点
- ✅ 简单高效，易于实现
- ✅ 保证最优前缀码
- ✅ 解码简单
- ❌ 需要两遍扫描（统计频率+编码）
- ❌ 符号集较大时效率下降
- ❌ 只能处理整数位编码

### 4.2 算术编码 (Arithmetic Coding)

#### 基本原理
- 将整个消息编码为一个[0, 1)区间内的单个小数
- 不受整数位限制，理论上可以达到熵极限

#### 编码过程

1. **初始化区间**: [0, 1)
2. **更新区间**:
   - 根据符号概率分割当前区间
   - 选择对应符号的子区间作为新区间
3. **输出编码**: 最终区间的任意值

#### 示例

符号概率: A(0.5), B(0.25), C(0.25)

消息: "AB"

```
初始区间: [0, 1)

处理A:
  A: [0, 0.5)
  B: [0.5, 0.75)
  C: [0.75, 1)
  选择A → 区间: [0, 0.5)

处理B:
  在[0, 0.5)内分割:
  A: [0, 0.25)
  B: [0.25, 0.375)
  C: [0.375, 0.5)
  选择B → 区间: [0.25, 0.375)

编码: 0.3 (区间[0.25, 0.375)中的任意值)
```

#### 优缺点
- ✅ 理论上可以达到熵极限
- ✅ 不需要整数位编码
- ✅ 适合自适应编码
- ❌ 计算复杂度高
- ❌ 需要高精度浮点运算
- ❌ 对计算精度敏感

### 4.3 LZ77算法

#### 基本原理
- 使用滑动窗口作为字典
- 用(距离, 长度)引用重复字符串
- 基于字符串匹配

#### 数据结构

```
┌─────────────────────────────────┐
│  滑动窗口                        │
├──────────┬──────────────────────┤
│  字典区  │      预读区           │
├──────────┼──────────────────────┤
│  已处理  │    待处理             │
│  的数据  │    的数据             │
└──────────┴──────────────────────┘
  ←────────── 32KB ──────────→
```

#### 编码格式

```
┌────────┬──────────┬────────┐
│ 长度   │  距离     │  字面  │
│ (8位)  │ (15位)    │  (8位) │
└────────┴──────────┴────────┘
```

- **长度(Length)**: 匹配字符串的长度
- **距离(Distance)**: 从当前位置回退的距离
- **字面(Literal)**: 未匹配的字节

#### 示例

原始数据: "ABCABCAB"

```
位置: 01234567

第1步: pos=0, 字符='A'
  查找: 字典为空
  输出: (Length=0, Distance=0, Literal='A')

第2步: pos=1, 字符='B'
  查找: 字典="A"
  输出: (Length=0, Distance=0, Literal='B')

第3步: pos=2, 字符='C'
  查找: 字典="AB"
  输出: (Length=0, Distance=0, Literal='C')

第4步: pos=3, 字符='A'
  查找: 字典="ABC"
  输出: (Length=0, Distance=0, Literal='A')

第5步: pos=4, 字符='B'
  查找: 字典="ABCA"
  输出: (Length=0, Distance=0, Literal='B')

第6步: pos=5, 字符='C'
  查找: 字典="ABCAB"
  匹配: "ABC" 距离=3, 长度=3
  输出: (Length=3, Distance=3)

第7步: pos=8, 字符='A'
  查找: 字典="ABCABC"
  匹配: "AB" 距离=3, 长度=2
  输出: (Length=2, Distance=3)

最终编码:
  A B C A B (3,3) (2,3)
```

#### 优缺点
- ✅ 适合压缩重复字符串
- ✅ 自适应性强
- ✅ 适合流式处理
- ❌ 匹配搜索开销大
- ❌ 对随机数据压缩效果差

### 4.4 LZ78算法

#### 基本原理
- 动态构建字典
- 每次匹配最长前缀
- 输出(字典索引, 下一字符)

#### 编码格式
```
┌──────────────┬────────┐
│  字典索引     │ 字符   │
└──────────────┴────────┘
```

#### 示例

字符串: "ABABACA"

```
初始字典: {空串: 0}

第1步: "A"
  最长匹配: 空串 (索引0)
  输出: (0, 'A')
  添加: "A" → 1
  字典: {0:"", 1:"A"}

第2步: "B"
  最长匹配: 空串 (索引0)
  输出: (0, 'B')
  添加: "B" → 2
  字典: {0:"", 1:"A", 2:"B"}

第3步: "A"
  最长匹配: "A" (索引1)
  输出: (1, 无字符)
  (特殊情况)

第4步: "B"
  最长匹配: "AB" (索引3) (第3步添加)
  ...
  
实际输出:
  (0,'A'), (0,'B'), (1,'A'), (1,'C'), (2,'A')
```

#### 优缺点
- ✅ 字典增长受限
- ✅ 编码简单
- ❌ 字典可能耗尽
- ❌ 不适合流式处理

### 4.5 LZW算法

LZW (Lempel-Ziv-Welch) 是LZ78的改进版本。

#### 改进点
- 移除字符输出，只输出字典索引
- 字典自动初始化为所有单字符

#### 示例

字符串: "TOBEORNOTTOBE"

```
初始化字典: {T:256, O:257, B:258, E:259, R:260, N:261}

输入: TOBEORNOTTOBE

1. 'T' → 输出: 256, W = 'T'
2. 'O' → 匹配 'O'? 否
   输出: 257, 添加 'TO':262, W = 'O'
3. 'B' → 输出: 258, 添加 'OB':263, W = 'B'
4. 'E' → 输出: 259, 添加 'BE':264, W = 'E'
5. 'O' → 匹配 'O'? 否
   输出: 257, 添加 'EO':265, W = 'O'
6. 'R' → 输出: 260, 添加 'OR':266, W = 'R'
7. 'N' → 输出: 261, 添加 'RN':267, W = 'N'
8. 'O' → 输出: 257, 添加 'NO':268, W = 'O'
9. 'T' → 匹配 'OT'? 否
   匹配 'TO'? 是 (262)
   下一个字符是'T'
   输出: 262, 添加 'OTT':269, W = 'T'
10. 'T' → 匹配 'TT'? 否
    匹配 'T'? 是
    输出: 256, 添加 'TT':270

输出: 256, 257, 258, 259, 257, 260, 261, 257, 262, 256
```

#### 优缺点
- ✅ 只输出索引，效率高
- ✅ 适合字典编码
- ✅ 广泛应用于GIF, TIFF
- ❌ 字典管理复杂
- ❌ 可能出现版权问题

## 五、混合压缩方案

### 5.1 DEFLATE算法

DEFLATE是zlib、gzip、PKZIP使用的压缩算法。

#### 算法组成

```
DEFLATE = LZ77 + 霍夫曼编码
         ↓          ↓
      字典编码    统计编码
```

#### 压缩流程

```
1. LZ77阶段
   ├─ 滑动窗口: 32KB
   ├─ 查找最长匹配
   └─ 输出(长度, 距离, 字面)

2. 霍夫曼编码阶段
   ├─ 构建三棵霍夫曼树
   │  ├─ 字面/长度树 (286个符号)
   │  ├─ 距离树 (30个符号)
   │  └─ 长度树 (19个符号)
   └─ 编码输出

3. 格式化输出
   ├─ zlib/gzip头部
   ├─ 压缩数据块
   └─ 校验和
```

#### 数据块类型

```
类型0: 未压缩块
类型1: 固定霍夫曼编码块
类型2: 动态霍夫曼编码块
类型3: 保留（未使用）
```

#### 优势
- ✅ LZ77去除重复模式
- ✅ 霍夫曼编码进一步压缩
- ✅ 适合多种数据类型
- ✅ 自适应性强

### 5.2 其他混合方案

| 方案 | 组合 | 应用 |
|------|------|------|
| LZMA | LZ77 + 算术编码 | 7z, xz |
| Bzip2 | Burrows-Wheeler + 霍夫曼 | .bz2 |
| RAR | LZ77 + 霍夫曼 + 其他 | .rar |
| Zstandard | LZ77 + 有限状态熵编码 | .zst |

## 六、压缩算法选择

### 6.1 选择标准

#### 数据类型
- **文本/代码**: 适合LZ77/LZ78类算法
- **图像**: 变换编码 + 量化
- **音频**: 预测编码 + 心理声学模型
- **视频**: 运动补偿 + 变换编码

#### 性能需求
- **速度优先**: LZ4, Snappy
- **压缩率优先**: LZMA, Zstandard
- **平衡**: zlib/DEFLATE

#### 使用场景
- **实时传输**: 快速压缩/解压
- **长期存储**: 高压缩率
- **嵌入式**: 低内存占用

### 6.2 常用压缩算法对比

| 算法 | 压缩率 | 速度 | 内存 | 应用 |
|------|--------|------|------|------|
| LZ4 | 低 | 极快 | 低 | Redis, Hadoop |
| Snappy | 低-中 | 快 | 低 | HBase, Cassandra |
| zlib | 中 | 中 | 中 | HTTP, PNG |
| Zstandard | 高 | 快 | 中 | 新项目推荐 |
| LZMA | 极高 | 慢 | 高 | 7z, xz |
| Bzip2 | 高 | 慢 | 高 | 传统Unix工具 |

## 七、压缩优化技巧

### 7.1 数据预处理

```
原始数据 → 预处理 → 压缩算法 → 压缩数据
           ↓
    提高压缩率
```

#### 常用预处理方法
- **Delta编码**: 对连续数值编码差值
- **行程编码 (RLE)**: 压缩连续相同值
- **位重组**: 优化数据排列
- **字符替换**: 替换为更高频字符
- **模式识别**: 提取固定模式

### 7.2 参数调优

#### zlib压缩级别
```c
Z_NO_COMPRESSION (0)    // 不压缩，速度最快
Z_BEST_SPEED (1)        // 最快速度，压缩率低
Z_DEFAULT_COMPRESSION (-1) // 默认，平衡
Z_BEST_COMPRESSION (9)  // 最高压缩率，速度慢
```

#### 窗口大小
```c
- 8-15位: 256B - 32KB
- 更大窗口 → 更高压缩率，更多内存
```

#### 内存级别
```c
- 1-9: 压缩内存分配级别
- 更高级别 → 更好压缩，更多内存
```

### 7.3 压缩策略

```c
Z_FILTERED      // 过滤数据（适合高度结构化数据）
Z_HUFFMAN_ONLY  // 仅霍夫曼编码（适合LZ77效率低的数据）
Z_RLE           // RLE + 霍夫曼（适合重复字节）
Z_FIXED         // 固定霍夫曼编码（速度快）
Z_DEFAULT_STRATEGY // 默认
```

## 八、实际应用中的考虑

### 8.1 压缩边界

#### 何时压缩有效
- 数据存在重复模式
- 符号分布不均匀
- 上下文相关性高

#### 何时压缩无效
- 已压缩数据（二次压缩反而增大）
- 加密数据（类似随机数据）
- 高熵数据（随机数、噪声）
- 数据过小（头部开销超过节省空间）

### 8.2 压缩阈值

```c
// 判断是否值得压缩
if (compressed_size >= original_size * 0.95) {
    // 压缩效果不佳，使用原始数据
    use_original_data();
}
```

### 8.3 多级压缩

```
原始数据 → 压缩1 → 加密 → 压缩2 → 最终输出
           ↓         ↓       ↓
      LZ77/DEFLATE AES  LZ4/快速压缩
```

**注意**: 加密后的数据熵值高，压缩效果差。

## 九、压缩效果评估

### 9.1 测试数据类型

```
1. 文本数据: 源代码、文档
2. 二进制数据: 可执行文件、数据库
3. 图像数据: 位图、RAW图像
4. 音频数据: WAV、PCM
5. 混合数据: 文档中嵌入图片
```

### 9.2 评估指标

```
压缩率 = 原始大小 / 压缩后大小
压缩比 = (1 - 压缩后大小 / 原始大小) × 100%
压缩速度 = 原始大小 / 压缩时间
解压速度 = 原始大小 / 解压时间
```

### 9.3 典型压缩效果

| 数据类型 | zlib压缩率 | 说明 |
|---------|-----------|------|
| 源代码 | 3-5:1 | 大量重复标识符 |
| 文本文件 | 2-4:1 | 英文/中文不同 |
| 可执行文件 | 2-3:1 | 某些部分已压缩 |
| 图像(BMP) | 2-10:1 | 取决于图像复杂度 |
| 随机数据 | 1:1 | 压缩无效 |
| 已压缩数据 | 0.9-1:1 | 二次压缩无效果 |

## 十、总结

### 关键概念

1. **熵**: 信息量的度量，压缩的理论极限
2. **冗余**: 数据中可被去除的重复信息
3. **编码**: 数据的表示方式，影响压缩效率
4. **字典编码**: LZ系列算法，基于字符串匹配
5. **统计编码**: 霍夫曼、算术编码，基于概率分布
6. **混合压缩**: 结合多种算法，如DEFLATE

### 压缩原则

1. **去除冗余**: 识别并消除各种类型的冗余
2. **优化编码**: 使用更紧凑的编码方式
3. **平衡性能**: 在压缩率、速度、内存之间权衡
4. **适配数据**: 根据数据特性选择合适的算法

### 学习路径

```
基础理论 → 简单算法 → 实际应用
   ↓          ↓          ↓
熵与冗余   霍夫曼      DEFLATE
信息论     LZ77        zlib
         LZW         gzip
```
